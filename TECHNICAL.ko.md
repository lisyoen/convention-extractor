# Convention Extractor — 기술 문서

## 1. 개요

Convention Extractor는 프로젝트 소스코드를 분석하여 코딩 컨벤션을 자동으로 추출하고, AI Code Agent가 바로 사용할 수 있는 `.rules` 파일을 생성하는 도구다.

### 전체 파이프라인 요약

1. 프로젝트 디렉토리에서 분석 대상 소스 파일을 수집한다.
2. 각 파일의 스타일 패턴을 LLM 없이 정적으로 분석한다.
3. 정적분석 결과를 기반으로 이상치(Outlier) 파일을 탐지하여 제외한다.
4. LLM에 소스코드와 정적분석 결과를 전달하여 컨벤션을 추출한다.
5. 채택률(adoption_rate) 기준으로 프로젝트 컨벤션을 확정한다.
6. 언어별 컨벤션 마크다운 파일을 생성한다.
7. (병합 모드) 기존 컨벤션 파일과 새 분석 결과를 자동 병합한다.
8. 전체 파일을 대상으로 컨벤션 준수 여부를 검사한다.

```
소스코드
  │
  ▼
[1. 파일 수집] ──→ (파일경로, 언어) 리스트
  │
  ▼
[2. 정적 분석] ──→ 파일별 스타일 프로파일 JSON
  │
  ▼
[3. 이상치 탐지] ──→ 이상치 파일 리스트
  │                        │
  │ (이상치 제외)           │ (컴플라이언스에서는 포함)
  ▼                        │
[4. LLM 컨벤션 추출] ──→ 컨벤션 분석 JSON
  │                        │
  ▼                        │
[5. 컨벤션 분류] ──→ accepted / rejected
  │                        │
  ▼                        │
[6. MD 생성] ──→ *_convention.md
  │                        │
  ▼ (--merge 사용 시)      │
[7. 컨벤션 병합] ──→ *_convention_merged.md
  │                        │
  ▼                        ▼
[8. 컴플라이언스 체크] ──→ refactoring_needed_*.txt
   (전체 파일 대상)
```

## 2. 처리 파이프라인

### 단계 1: 파일 수집 (File Collection)

프로젝트 디렉토리를 재귀 탐색하여 분석 대상 소스 파일을 수집한다.

**언어별 확장자 매핑:**

| 언어 | 확장자 |
|------|--------|
| C/C++ | `.c`, `.h`, `.cpp`, `.hpp` |
| Python | `.py` |
| JavaScript | `.js`, `.jsx` |
| TypeScript | `.ts`, `.tsx` |
| Java | `.java` |
| Go | `.go` |
| Rust | `.rs` |

**제외 규칙:**
- 기본 제외: `.git`, `node_modules`, `build`, `dist`, `__pycache__`, `vendor` 등 빌드/의존성 디렉토리
- 사용자 정의: 프로젝트 루트의 `.convention-ignore` 파일 (`.gitignore` 문법)

**대규모 프로젝트 처리:**
`max_files` 설정값을 초과하는 프로젝트의 경우, 언어별 균등 샘플링(equal sampling)을 수행한다. 예를 들어 `max_files=100`이고 3개 언어가 있으면 언어당 약 33개 파일을 무작위 선택한다. 이를 통해 특정 언어에 편향되지 않은 분석이 가능하다.

> **입력**: 프로젝트 루트 경로  
> **출력**: `(파일경로, 언어)` 튜플 리스트

---

### 단계 2: 정적 분석 (Static Analysis)

LLM을 사용하지 않고 기계적으로 추출 가능한 코드 스타일 패턴을 수집한다. 이 단계의 결과는 두 가지 용도로 사용된다:

1. **이상치 탐지의 비교 기준** (단계 3)
2. **LLM 프롬프트의 ground truth** (단계 4) — LLM이 "감"이 아닌 실측 데이터를 기반으로 판단하도록 유도

**분석 항목:**

| 항목 | 설명 | 예시 값 |
|------|------|---------|
| 들여쓰기(Indentation) | 탭/스페이스, 칸 수 | `spaces_4: 92%`, `tabs: 8%` |
| 네이밍 패턴(Naming) | snake_case, camelCase, PascalCase 비율 | `camelCase: 85%`, `PascalCase: 15%` |
| 세미콜론(Semicolon) | 사용/미사용 비율 | `with: 97%`, `without: 3%` |
| 따옴표(Quote Style) | 작은따옴표/큰따옴표 비율 | `single: 78%`, `double: 22%` |
| 브레이스 스타일(Brace) | K&R, Allman 등 | `kr_style: 95%` |
| 라인 길이(Line Length) | P90 기준 최대 길이 | `p90: 98` |
| 변수 선언(Variable) | var/const/let 비율 (JS/TS) | `const: 72%`, `let: 25%`, `var: 3%` |

각 파일에 대해 **파일별 스타일 프로파일**을 생성한다. 개별 파일의 스타일 특성을 기록하여 이상치 탐지의 기반을 마련한다.

> **입력**: 수집된 파일의 소스코드 내용  
> **출력**: 파일별 스타일 프로파일 JSON (각 항목의 비율 정보 포함)

---

### 단계 3: 이상치 탐지 (Outlier Detection)

같은 언어의 파일들 사이에서 프로젝트 컨벤션을 따르지 않는 **이상치 파일**을 탐지한다. 이상치는 외부 라이브러리 복사본, 레거시 코드, 자동 생성 코드 등 프로젝트의 주류 스타일과 다른 파일이다.

**왜 컨벤션 추출 전에 이상치를 판별할 수 있는가?**

이 단계는 LLM이 추출하는 고수준 컨벤션(디자인 패턴, 에러 처리 방식 등)이 아니라, 정적분석(단계 2)에서 수집한 **기계적으로 측정 가능한 스타일 지표**만으로 이상치를 판별한다. 들여쓰기, 네이밍, 세미콜론 등의 정적 지표는 LLM 없이도 다수결로 비교할 수 있다. 이 단계의 목적은 LLM 분석의 입력 품질을 높이는 것이므로, 정적 지표만으로도 충분하다.

**비교 항목 (5개):**

1. 들여쓰기 스타일 (Indentation)
2. 네이밍 패턴 (Naming Convention)
3. 변수 선언 스타일 (Variable Declaration)
4. 세미콜론 사용 (Semicolon Usage)
5. JSDoc/주석 스타일 (Documentation Style)

**판정 방식:**
- 같은 언어 파일들의 각 항목에 대해 **다수결(majority)**을 산출
- 개별 파일이 다수결과 불일치하는 항목 수를 카운트
- `outlier_threshold`(기본 3) 이상 불일치 시 이상치로 판정

**최소 파일 수 요건:**
같은 언어의 파일이 3개 미만이면 이상치 탐지를 건너뛴다. 2개면 동점이 발생할 수 있고, 1개는 비교 대상이 없기 때문이다. 3개 이상일 때만 다수결이 의미 있다.

**`outlier_threshold`가 3인 이유:**
현재 비교 항목이 5개이므로 과반(3/5)을 기준으로 한다. 비교 항목이 늘어나면 threshold도 조정해야 하지만, 현재 5개 항목은 코드 스타일의 핵심 지표를 모두 포함하고 있어 추가 항목의 필요성은 낮다. 필요 시 config에서 조정 가능.

**이상치 파일의 처리:**

| 단계 | 이상치 포함 여부 | 이유 |
|------|----------------|------|
| LLM 컨벤션 추출 (단계 4) | **제외** | 컨벤션 추출 결과의 오염 방지 |
| 컴플라이언스 체크 (단계 7) | **포함** | 위반 파일로 탐지하여 리팩토링 대상 식별 |

**왜 이상치를 LLM 분석에서 제외하는가?**

이상치 파일은 프로젝트의 주류 스타일과 크게 다른 코드다. 이런 파일이 LLM 분석에 포함되면 LLM이 소수 파일의 스타일을 프로젝트 전체의 컨벤션으로 오인할 수 있다. 예를 들어 11개 파일이 2칸 들여쓰기인데 1개만 4칸이면, LLM이 이 1개에 영향받아 "4칸이 표준"이라고 잘못 추출할 수 있다. 이상치를 제외함으로써 LLM이 다수 파일의 패턴에만 집중하여 정확한 컨벤션을 추출한다. 단, 컴플라이언스 체크에서는 이상치를 포함하여 "어떤 파일이 컨벤션을 따르지 않는지" 보고한다.

> **입력**: 파일별 스타일 프로파일 (단계 2의 출력)  
> **출력**: 이상치 파일 리스트

---

### 단계 4: LLM 컨벤션 추출 (LLM Convention Extraction)

정적 분석으로 파악할 수 없는 고수준 코딩 패턴(코드 구조, 에러 처리 방식, 디자인 패턴 등)을 LLM을 통해 추출한다.

**배치 처리:**
- 이상치를 제외한 소스 파일을 `batch_size`(기본 5) 단위로 묶어 LLM에 전송
- 각 배치의 프롬프트에는 해당 파일들의 **정적분석 요약(비율 포함)**이 ground truth로 포함됨
- LLM은 정적분석 수치를 참고하면서 코드를 읽고 컨벤션을 추출

**추출 카테고리:**

| 카테고리 | 예시 |
|----------|------|
| 네이밍(Naming) | 함수명 camelCase, 상수 UPPER_SNAKE_CASE |
| 포매팅(Formatting) | 들여쓰기 2칸, 한 줄 최대 100자 |
| 코드 구조(Structure) | early return 패턴, 함수 길이 제한 |
| 주석(Comments) | JSDoc 필수, 한국어/영어 혼용 |
| 패턴(Patterns) | async/await 선호, 에러 처리 방식 |

**채택률(adoption_rate) 부여:**
LLM은 각 컨벤션에 0~100 사이의 채택률을 부여한다. 이 값은 "분석 대상 파일 중 몇 %가 이 패턴을 따르고 있는가"를 의미한다.

**다중 배치 병합(Merge):**
배치가 2개 이상일 경우, 이전 배치의 분석 결과를 다음 배치의 프롬프트에 포함하여 누적 분석한다. LLM은 기존 컨벤션의 채택률을 업데이트하거나 새로운 컨벤션을 추가한다.

**Thinking 모델 지원:**
Qwen3 등 thinking 모델 사용 시 자동 감지하여 `max_tokens`와 `timeout`을 보정하고, 응답에서 `<think>` 블록을 자동 제거한다.

> **입력**: 소스코드 배치 + 정적분석 요약 JSON  
> **출력**: 컨벤션 분석 JSON (카테고리별 `description` + `adoption_rate`)

---

### 단계 5: 컨벤션 분류 (Convention Classification)

LLM이 추출한 컨벤션을 채택률 기준으로 분류한다.

**분류 기준:**

```
adoption_rate ≥ adoption_threshold(기본 90%)  →  accepted (프로젝트 컨벤션)
adoption_rate < adoption_threshold             →  rejected (참고용)
```

- **accepted**: 프로젝트의 공식 코딩 컨벤션으로 확정. `.rules` 파일과 컴플라이언스 체크의 기준이 됨
- **rejected**: 일부 파일에서만 사용되는 패턴. `conventions.json`에 기록되나 `.rules` 파일에는 포함되지 않음

예를 들어 `adoption_threshold=90`일 때:
- "함수명은 camelCase" (adoption_rate: 97) → **accepted**
- "에러 메시지는 한국어" (adoption_rate: 62) → **rejected**

> **입력**: 컨벤션 분석 JSON (단계 4의 출력)  
> **출력**: accepted/rejected로 분류된 컨벤션

---

### 단계 6: 컨벤션 MD 생성 (Convention Markdown Generation)

확정된 컨벤션을 언어별 마크다운 파일로 생성한다.

**생성 파일:** `{언어}_convention.md` (예: `javascript_convention.md`, `python_convention.md`)

**파일 구조:**
- 카테고리별 컨벤션 규칙 나열
- 각 규칙에 **Good** (권장) / **Avoid** (지양) 예시 포함
- Continue, Cline 등 AI Code Agent의 `.rules` 파일로 바로 사용 가능한 포맷

이 파일을 프로젝트의 `.continue/rules/` 또는 `.cline/rules/` 디렉토리에 배치하면 AI Code Agent가 코드 생성 시 프로젝트 컨벤션을 자동으로 준수한다.

> **입력**: accepted로 분류된 컨벤션  
> **출력**: `javascript_convention.md`, `python_convention.md` 등 언어별 파일

---

### 단계 7: 컨벤션 병합 (Convention Merge) — v1.7

`--merge <기존_컨벤션_파일.md>` 옵션 사용 시 실행되는 단계. 기존에 확정된 컨벤션 파일과 새로 분석한 결과를 자동으로 병합한다.

**기존 컨벤션 파일 파싱:**
- 마크다운 구조에서 `### 카테고리` → `- **키**: 값` 패턴을 인식
- 카테고리/키/값 구조의 딕셔너리로 변환

**병합 로직:**
- 기존 항목과 새 분석 결과의 키를 정규화(대소문자, 공백/언더스코어 통일)하여 매칭
- **유지**: 기존과 동일한 값 → 그대로 보존
- **신규**: 새 분석에만 있는 항목 → `[신규]` 태그와 함께 추가
- **변경**: 기존과 다른 값 → 기존 값을 유지하되 `[변경] 기존: X → 신규: Y` diff 표시
- 기존에만 있고 새 분석에 없는 항목 → 유지 (삭제하지 않음)

**출력:**
- `*_convention_merged.md`: 병합 요약 테이블 + 전체 컨벤션 (태그 포함)
- `*_convention.md`: 새 분석 결과만의 MD (기존과 동일, 항상 생성)

**왜 기존 항목을 우선 보존하는가?**

기존 컨벤션 파일은 팀에서 리뷰하고 확정한 규칙이다. 새 분석 결과는 모델이나 설정이 달라질 수 있으므로, 기존 확정 항목을 자동으로 덮어쓰지 않는다. 변경 사항은 diff로 표시하여 사람이 최종 판단할 수 있도록 한다.

> **입력**: 기존 컨벤션 MD 파일 + 새 분석의 accepted 컨벤션  
> **출력**: `*_convention_merged.md` (병합 요약 + 태그 포함 컨벤션)

---

### 단계 8: 컴플라이언스 체크 (Compliance Check)

추출된 컨벤션을 기준으로 프로젝트의 **모든 파일**(이상치 포함)을 검사한다.

**처리 방식:**
- `compliance_batch_size` 단위로 소스코드와 확정 컨벤션 JSON을 LLM에 전송
- LLM은 각 파일에서 컨벤션 위반 항목만 식별 (적합 항목은 보고하지 않음)
- 이상치 파일의 위반은 `[이상치]` 태그를 붙여 구분

**출력 형식:**
파일별 `refactoring_needed_*.txt`에 위반 목록이 기록된다. 위반이 없는 파일은 출력에 포함되지 않는다.

> **입력**: 전체 소스코드 + 확정된 컨벤션 JSON  
> **출력**: `refactoring_needed_*.txt` (파일별 위반 목록, `[이상치]` 태그 포함)

## 3. 설정 영향도

| 설정 | 기본값 | 효과 |
|------|--------|------|
| `adoption_threshold` | 90 | 컨벤션 확정 기준. **높이면**(95) 거의 모든 파일이 따라야 인정 → 엄격. **낮추면**(70) 다수결만으로 인정 → 관대 |
| `batch_size` | 5 | LLM 컨벤션 추출 배치 크기. **크면** 처리 빠르지만 LLM 컨텍스트 한계로 품질 저하 가능. **작으면** 정확하지만 배치 수 증가로 느려짐 |
| `compliance_batch_size` | — | 컴플라이언스 체크 배치 크기. `batch_size`와 동일한 트레이드오프. 정확도가 중요하면 작게 설정 |
| `max_files` | 1000 | 초과 시 언어별 균등 샘플링 발동. 대규모 프로젝트에서 분석 시간과 비용 제어 |
| `outlier_threshold` | 3 | 현재 비교 항목 5개 기준으로 과반(3/5). **높이면**(4~5) 극단적 파일만 이상치. **낮추면**(2) 적극적으로 이상치 제외. 비교 항목 추가 시 조정 필요하나, 핵심 스타일 지표 5개로 충분하여 변경 가능성 낮음 |
| `--merge` (CLI) | — | 기존 컨벤션 MD 파일 경로. **지정 시** 기존 컨벤션과 새 분석 결과를 병합하여 `*_convention_merged.md` 생성. **미지정 시** 기존 동작과 100% 동일 (하위 호환) |

## 4. 출력 파일 요약

| 파일 | 생성 단계 | 용도 |
|------|----------|------|
| `*_convention.md` | 6단계 (MD 생성) | AI Code Agent `.rules` 파일로 직접 사용 |
| `*_convention_merged.md` | 7단계 (병합) | 기존 컨벤션과 병합된 결과. `--merge` 사용 시에만 생성 |
| `conventions.json` | 4~5단계 (추출/분류) | 전체 분석 결과. accepted/rejected 포함, 병합 정보(`merge_info`) 포함 |
| `refactoring_needed_*.txt` | 8단계 (컴플라이언스) | 컨벤션 위반 파일 목록. 리팩토링 대상 식별용 |
| `extract_convention_result_*.log` | 전 단계 | 실행 로그. 단계별 소요시간, 처리 파일 수 등 기록 |
| `debug_*.log` | 에러 발생 시 | LLM 응답 원문 등 디버그 정보. 정상 실행 시 미생성 |

## 5. 설계 의도

**왜 정적 분석과 LLM을 함께 사용하는가?**

정적 분석은 들여쓰기, 네이밍, 세미콜론 등 기계적으로 측정 가능한 패턴을 정확한 비율로 제공한다. 반면 "early return 선호", "에러 처리는 try-catch로 감싸기" 같은 고수준 패턴은 LLM만이 파악할 수 있다. 정적분석 결과를 LLM 프롬프트에 ground truth로 제공함으로써, LLM이 측정 가능한 항목에서 잘못된 판단을 내리는 것을 방지하고 고수준 분석에 집중하도록 유도한다.

**왜 이상치를 탐지하는가?**

외부 라이브러리 복사본, 자동 생성 코드, 레거시 파일 등은 프로젝트의 주류 스타일과 다를 수 있다. 이런 파일이 LLM 분석에 포함되면 컨벤션 추출 결과가 오염된다. 이상치 탐지는 이러한 노이즈를 사전에 제거하여 추출 품질을 높인다. 동시에 컴플라이언스 체크에서는 이상치를 포함하여 "어떤 파일이 컨벤션을 따르지 않는지" 전체 그림을 제공한다.
